# 工程师抗拒被"蒸馏"，企业的Skills从何而来？五大招破局

## 编译摘要

### 1. 浓缩
- **核心结论1**: Skills 沉淀制造了委托-代理困境——企业希望工程师蒸馏知识，但工程师清楚这等于训练替代品
  - 关键证据: 理性博弈导致消极抵抗、质量投毒、上下文污染三种对抗策略
- **核心结论2**: Skills 体系的瓶颈不是技术生命周期，而是“谁愿意启动这个生命周期”的制度问题。
  - 关键证据: 文章指出，Agentic Skills 可以被发现、实践、蒸馏、存储、组合、评估和更新，但如果工程师的个人最优策略是隐瞒或投毒，生命周期不会自然运转。
- **核心结论3**: 破局需要把“被蒸馏”改写成“拥有知识资产的收益权、声望权和上升路径”。
  - 关键证据: 五个制度方案分别对应持续分润（Skills 版税制）、职业跃迁（创建 Skills 是晋升阶梯）、低主动摩擦（系统自动观察 + 工程师审核）、同侪声望（Skills 竞技场）和根本安全感（保护期、转岗优先权、退休金/买断）。
- **核心结论4**: 防投毒不能只靠激励，还必须有技术纵深。
  - 关键证据: 文章提出形式化验证、血缘追踪、对抗性测试、灰度发布和多源交叉验证，用来防止 Skill 中的隐藏缺陷、上下文遗漏和恶意后门。

### 2. 质疑
- **关于"版税制"的质疑**: 版税计算依赖调用计量系统，技术实现复杂度可能抵消激励效果；且"复杂度系数"如何量化？
- **关于"角色跃迁制"的质疑**: 前提是企业有足够高阶工作吸纳释放的工程师，如果企业只是在降本，承诺不可信
- **关于“对抗性蒸馏”的质疑**: 自动观察可以降低主动配合门槛，但也可能制造监控感和不信任感。若没有透明边界，它会加剧工程师防御，而不是促成共享。
- **关于“组织保险”的质疑**: 保护期、转岗优先权和知识买断若不写入真实合同，只是口号。即使写入，也要解决企业调整、绩效评价和 Skill 归属的法律细节。
- **关于数据可靠性的质疑**: 文章提供的是制度设计草案，不是已验证案例。它适合进入 research agenda 或 topic 讨论，但不应直接当作企业最佳实践。
- **关于 Skill 质量的质疑**: 激励工程师创建更多 Skills 不等于得到高质量 Skills。组织还需要评测集、调用日志、失败归因、版本治理和淘汰机制，否则版税和排行榜会奖励数量而非可靠性。

### 3. 对标
- **对标开源软件**: 文章最有力的类比是 Linux。Linus “蒸馏”了自己的操作系统能力，但获得的是长期影响力和裁判权，而不是被替代。这说明知识外化不必然削弱个人价值，关键在权利结构。
- **对标 Tacit Knowledge Lock-In**: 企业想提取工程师隐性知识，本质上是在争夺 [[Tacit-Knowledge-Lock-In|隐性知识锁定]] 的归属：知识留在人身上，还是沉淀为组织资产。
- **对标 Knowledge Compilation**: Skill 不是普通文档，而是可执行知识编译产物。它比 wiki 页面更接近“组织记忆 + 自动执行”的交叉层。
- **迁移到 Agentic Work Atlas**: 本文应支撑一个 topic 问题：AI-ready organization 不只是能不能写 Skills，而是组织是否有制度让高价值 Skills 自愿、可信、可审计地涌现。
