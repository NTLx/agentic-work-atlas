# 20260410-better-code

## 编译摘要

### 1. 浓缩
- **核心结论1**: 用 AI 助手交付更差代码是一种选择，我们也可以选择交付更好的代码
  - 关键证据: 如果编码助手降低了代码质量，应该直接解决问题而非忽视
- **核心结论2**: 编码助手可以处理简单但耗时的重构任务，成本降至几乎为零
  - 关键证据: API 设计修复、命名清理、功能合并、文件拆分等任务可委托给 AI
- **核心结论3**: 复合工程（Compound Engineering）让每次项目后的有效做法沉淀到未来指令中
  - 关键证据: 每个项目以回顾结束，记录有效内容供未来助手运行使用

### 2. 质疑
- **关于"更好代码"的质疑**: "更好"的定义是什么？可读性？性能？可维护性？不同目标可能冲突
- **关于重构的质疑**: AI 做的重构如何验证正确性？大规模重构的风险如何控制？
- **关于复合工程的质疑**: 指令迭代是否有天花板？过度定制的指令是否丧失通用性？

### 3. 对标
- **跨域关联1**: 类似技术债务的"预防优于治疗"——最好的技术债务缓解措施是一开始就不承担它
- **跨域关联2**: 类似知识管理中的 LLM Wiki——信息提前编译 vs 运行时查询
- **可迁移场景**: 任何需要持续改进的领域——代码、文档、设计系统、流程
