# What Is Code?

## 编译摘要

### 1. 浓缩
- **核心结论1**: 代码不仅是给机器的指令，更是问题领域的概念模型（Conceptual Model）。
  - 关键证据: LLM 使得指令生成变得廉价，但代码作为理解系统的中介（中介人类与机器、人类与工具、人类与人类）的价值反而提升。
- **核心结论2**: 编码本质上是构建词汇表（Vocabulary Building）的过程。
  - 关键证据: 开发者通过将领域词汇映射到技术领域，在代码中发现并塑造概念、命名、边界和关系。
- **核心结论3**: 编程语言是思考工具（Thinking Tools），而非仅是表达工具。
  - 关键证据: 不同的语言约束（如 Rust 的所有权、Go 的 channel）迫使开发者以特定方式看待结构和权衡，主动编写代码是深层思考的一部分。

### 2. 质疑
- **关于"代码是词汇表"的质疑**: 如果 LLM 可以理解非结构化的意图并直接生成端到端的系统，那么在中间层维护一套精确的"代码词汇表"是否仍然必要？是否存在一种可能，即"词汇表"最终会演变为纯自然语言或更高层的 DSL，而传统的"代码"退化为不可读的中间产物？
- **关于"认知债务"的质疑**: 文章提到 LLM 产生的代码如果未被团队理解会积累认知债务。但如果这部分代码是自包含的、经过充分测试的，且生命周期极短（随时可以重写），那么理解其"词汇"的必要性是否降低了？
- **数据可靠性**: 文章主要基于作者（Unmesh Joshi）的经验和软件工程的历史趋势（如 DDD, Agile），属于思辨性论述而非实验性研究。

### 3. 对标
- **跨域关联1**: 类似于**建筑学中的图纸**。图纸既是工人的施工指南（机器指令），也是建筑师与业主沟通的概念模型。随着 3D 打印技术的成熟，手动画施工图可能变得不重要，但定义空间逻辑和词汇（如"承重墙"、"流线"）依然核心。
- **跨域关联2**: 类似于**数学符号体系**。数学符号不仅是记录结果的工具，它们塑造了人类处理抽象问题的思维方式（例如微积分符号对物理学发展的推动）。
